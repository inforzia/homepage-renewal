# 외부 참고자료 기반 페이지 구현 스킬

## 목적

본 스킬은 외부 웹사이트, 디자인 시안, 이미지, Figma 화면, 기존 서비스 화면, HTML/CSS 자료 등 제공받은 참고자료를 근거로 웹 페이지를 구현하기 위한 작업 절차를 정의한다.

이 스킬의 목표는 참고자료를 시각적으로 추정하여 유사하게 만드는 것이 아니라, 확인 가능한 정보에 근거하여 화면 구조, 컴포넌트 구성, 스타일, 반응형 동작 및 인터랙션을 최대한 정확하게 재현하는 것이다.

작업 과정과 구현 근거는 `TODO.md`에 기록하며, 사용자의 최종 승인 전까지 작업을 완료 처리하지 않는다.

`TODO.md`는 단순 진행 현황 요약이 아니라, 어떤 대상을 어떤 스타일 규칙으로 구현했는지와 검증 결과가 무엇이었는지를 나중에 문서만 읽고도 재구성할 수 있을 정도로 구체적으로 기록해야 한다.

---

## 적용 범위

다음과 같은 작업에 적용한다.

- 외부 웹사이트 화면 클론 구현
- Figma 또는 이미지 시안 기반 페이지 구현
- 기존 서비스 화면의 신규 프로젝트 이식
- 기존 페이지의 디자인 리뉴얼
- 특정 페이지 또는 섹션의 UI 재현
- 참조 페이지를 기반으로 한 반응형 화면 구현
- 제공된 HTML, CSS, 디자인 가이드 기반 화면 개발

단, 단순 텍스트 수정, 단순 색상 변경, 명확한 단일 스타일 수정처럼 별도 분석과 계획이 불필요한 작업에는 적용하지 않아도 된다.

## 핵심 원칙

### 1. 추정 대신 확인 가능한 근거를 우선한다

스타일 값, 요소 구조, 여백, 폰트, 색상, 반응형 규칙, 인터랙션은 가능한 한 참고자료에서 직접 확인한 사실을 기준으로 구현한다.

다음 우선순위를 따른다.

1. 브라우저의 실제 computed style
2. DOM 구조 및 실제 렌더링 결과
3. 디자인 시안의 속성값
4. 프로젝트 내 기존 디자인 시스템 또는 공통 컴포넌트
5. 화면 이미지 및 시각적 추론

이미지 또는 제한된 자료만 제공되어 정확한 값을 확인할 수 없는 경우에는 추정값임을 `TODO.md`에 기록한다.

기록 시에는 `대략 비슷하게`, `원본에 더 가깝게`, `적당히 조정` 같은 모호한 표현을 피하고, 실제 확인한 값 또는 확인 불가 사실을 그대로 적는다.

---

### 2. 코드 선언값이 아닌 최종 렌더링 결과를 기준으로 판단한다

CSS 파일 또는 컴포넌트 코드에 선언된 값만으로 구현 정확도를 판단하지 않는다.

다음과 같은 최종 적용 값을 우선 확인한다.

- `font-family`
- `font-size`
- `font-weight`
- `font-style`
- `line-height`
- `letter-spacing`
- `word-spacing`
- `color`
- `background-color`
- `border`
- `border-radius`
- `box-shadow`
- `width`
- `min-width`
- `max-width`
- `height`
- `min-height`
- `padding`
- `margin`
- `gap`
- `display`
- `position`
- `top`, `right`, `bottom`, `left`
- `align-items`
- `justify-content`
- `grid-template-columns`
- `flex-direction`
- `overflow`
- `z-index`
- `opacity`
- `transform`
- `transition`
- `animation`

특히 텍스트 크기, 굵기, 줄 높이, 여백, 간격은 화면 유사도에 큰 영향을 주므로 반드시 검증한다.

`TODO.md`에는 가능하면 다음 형식으로 기록한다.

```md
- Hero 제목: `styles.heroTitle`, `font-size: 64px`, `line-height: 84px`, `letter-spacing: -1.6px`
- CTA 버튼: `styles.heroAction`, `min-height: 44px`, `padding: 12px 20px 12px 24px`, `border-radius: 9999px`
```

### 3. 기존 구현물과 컴포넌트 재사용을 우선한다

새로운 페이지를 구현하기 전에 프로젝트 내부에 이미 구현된 관련 컴포넌트, 레이아웃, 스타일 규칙이 있는지 반드시 확인한다.

우선 확인할 대상은 다음과 같다.

- 동일하거나 유사한 페이지
- 공통 Header, Footer, Page Layout
- 카드, 버튼, 탭, 입력 필드, 모달 등 기존 UI 컴포넌트
- 페이지 컨테이너, Grid, 반응형 레이아웃 유틸리티
- 색상, 타이포그래피, 간격, 브레이크포인트 토큰
- 기존 아이콘, 이미지, 공통 스타일 클래스
- 유사한 데이터 목록 및 항목별 동작 처리 로직

기존에 구현된 관련 컴포넌트가 있다면, 새로 동일한 구조를 만들기보다 해당 컴포넌트를 재사용하거나 필요한 범위에서 확장하는 방향으로 구현한다.

새 컴포넌트를 만드는 것은 기존 구현으로 요구사항을 충족할 수 없거나, 확장 시 기존 컴포넌트의 책임과 API가 과도하게 복잡해지는 경우에만 검토한다.

단일 페이지 안에서 반복되는 요소는 반복된다는 이유만으로 별도 컴포넌트로 분리하지 않는다. 다음 두 조건을 **모두** 충족하는 경우에만 별도 컴포넌트 분리를 검토한다.

1. 해당 요소가 현재 화면의 여러 위치 또는 다른 화면에서도 재사용된다.
2. 해당 요소가 내부에서 관리해야 하는 고유한 상태값 또는 독립적인 동작 책임을 가진다.

예를 들어, 단순히 같은 카드 모양이나 같은 제목 스타일이 반복되는 경우에는 별도 컴포넌트를 만들지 않는다. `module.css`의 공통 클래스와 `className` 조합으로 스타일을 재사용한다.

반복 항목마다 서로 다른 데이터에 따라 클릭, 선택, 이동, 표시 등의 동작만 달라지는 경우에도 항목별 컴포넌트를 불필요하게 분리하지 않는다. 데이터 기반 렌더링과 공통 함수 또는 이벤트 핸들러를 사용하여 처리한다.

예시:

```jsx
const Page = () => {
    const handleFeatureClick = (featureId: string) => {
        // 항목별 데이터에 따라 공통 동작 처리
    };

    return features.map((feature) => (
        <button
            key={feature.id}
            className={styles.featureCard}
            onClick={() => handleFeatureClick(feature.id)}
        >
            <strong className={styles.featureTitle}>{feature.title}</strong>
            <p className={styles.featureDescription}>{feature.description}</p>
        </button>
    ));
};
```

다음과 같은 경우에는 별도 컴포넌트 분리를 피한다.

- 한 페이지 안에서만 사용되는 단순한 반복 마크업
- 스타일 외에 독립적인 책임이 없는 요소
- props가 지나치게 많아져 구조가 오히려 복잡해지는 경우
- 단순 텍스트, 아이콘, 제목, 카드 레이아웃의 반복
- 데이터와 이벤트 핸들러만 다른 목록 항목
- 별도 컴포넌트로 분리해도 테스트, 재사용성, 상태 관리 측면의 이점이 없는 경우

다음과 같은 경우에는 별도 컴포넌트 분리를 검토한다.

- 여러 페이지 또는 화면의 여러 영역에서 같은 구조와 동작을 반복 사용한다.
- 내부 열림/닫힘, 선택, 입력, 로딩, 애니메이션 등 독립적인 상태를 관리한다.
- 복잡한 이벤트 처리나 접근성 로직을 자체적으로 가진다.
- 자체 테스트 단위로 분리할 실질적인 필요가 있다.
- 부모 컴포넌트의 책임을 과도하게 증가시키는 독립 기능 단위다.

기존 컴포넌트를 재사용하지 않고 새로 구현하는 경우에는 그 이유를 `TODO.md`에 간단히 기록한다.

---

### 4. 기존 프로젝트 구조와 디자인 시스템을 우선 존중한다

새로운 페이지를 구현할 때는 먼저 프로젝트에 이미 존재하는 다음 요소를 확인한다.

- 공통 레이아웃
- 디자인 토큰
- 색상 변수
- 타이포그래피 규칙
- 버튼, 카드, 입력창 등 공통 UI 컴포넌트
- 아이콘 라이브러리
- 스타일링 방식
- 반응형 브레이크포인트
- 상태 관리 방식
- 라우팅 구조
- API 호출 방식

유사한 기능이나 화면이 이미 구현되어 있다면 새로 만들기 전에 재사용 또는 확장 가능성을 검토한다.

## 작업 시작 전 확인 사항

작업을 시작하기 전에 다음 정보를 확인한다.

- 구현 대상 페이지 또는 섹션
- 참고자료의 종류
    - 웹사이트 URL
    - 디자인 시안
    - 이미지
    - Figma 링크 또는 화면
    - 기존 구현 코드
    - HTML/CSS 자료

- 적용할 프로젝트와 경로
- 구현 범위
    - 정적 화면만 필요한지
    - 인터랙션이 필요한지
    - API 연동이 필요한지
    - 반응형 구현이 필요한지

- 지원해야 하는 화면 크기 또는 브라우저
- 기존 컴포넌트 및 디자인 시스템 사용 여부
- 사용자에게 확인이 필요한 불명확한 요구사항

명확하지 않은 항목이 있더라도 작업을 중단하지 않는다. 확인 가능한 범위부터 분석 및 구현하고, 불확실한 부분은 가정으로 기록한다.

단, 별도 계획이 필요한 작업이라면 분석 후 바로 구현에 들어가지 않고 우선 구현 계획을 문서화한 뒤, 작업 시작 전에 그 계획 내용을 사용자에게 확인받는다.

## 작업 순서

## 1. 사전 준비

작업 시작 시 해당 작업 범위에 대한 `TODO.md` 파일을 생성하거나 기존 파일을 갱신한다.

파일은 작업 대상과 가까운 위치에 두되, 프로젝트의 기존 TODO 관리 규칙이 있다면 그 규칙을 따른다.

모든 작업이 완료된 TODO.md 파일들은 별도의 폴더로 이동되며, 작업일시 + 작업내용 형태의 이름을 가짐으로써 나중에 추적하기 용이하게끔 한다.

### TODO.md 기본 형식


### 사전 준비 시 준수 사항

- 작업 단위가 명확하도록 작업명을 작성한다.
- 구현 대상 페이지, 경로, 참고자료를 기록한다.
- 참고자료가 여러 개인 경우 우선순위를 기록한다.
- 이미 구현된 유사 화면 또는 재사용 후보 컴포넌트를 기록한다.
- 확인하지 못한 내용은 추정 또는 가정으로 명시한다.
- 아직 구현하지 않은 항목을 완료 상태로 표시하지 않는다.
- 구현 또는 수정 대상이 되는 요소 이름을 명시한다.
- 각 요소에 적용할 스타일 규칙 또는 컴포넌트 재사용 방식을 기록한다.
- 스타일 규칙은 가능하면 클래스명 또는 컴포넌트명과 함께 적는다.

## 2. 참고자료 분석

참고자료를 분석하여 페이지의 구조, 구성 요소, 스타일 규칙, 동작 방식을 정리한다.

분석 단계에서는 즉시 코드를 작성하기보다, 구현에 필요한 정보를 먼저 수집하고 구조화한다.

### 2.1 페이지 전체 구조 분석

다음 항목을 파악한다.

- 페이지가 어떤 상위 영역으로 구성되는지
- 각 영역의 표시 순서
- 화면 전체의 최대 너비와 좌우 여백
- 헤더 및 푸터의 존재 여부
- 스크롤 구조
- 고정 요소 또는 sticky 요소의 존재 여부
- 배경색, 배경 이미지, 그라데이션
- 데스크톱과 모바일에서의 구조 차이

예시:

```md
- Header
- Main Visual
- 서비스 소개 섹션
- 기능 카드 섹션
- 사용 절차 섹션
- CTA 섹션
- Footer
```

---

### 2.2 섹션별 구조 분석

각 섹션에 대해 다음을 정리한다.

- 섹션의 역할
- 제목 및 설명 구조
- 내부 콘텐츠 수
- 콘텐츠 정렬 방식
- 카드 또는 목록 구조
- 이미지, 아이콘, 버튼의 존재 여부
- 섹션 간 여백
- 배경색 또는 구분선
- 데스크톱과 모바일에서의 배치 변경 여부
- 구현 시 사용할 클래스 또는 컴포넌트 후보

예시:

```md
### 기능 소개 섹션

- 섹션 제목 1개
- 보조 설명 1개
- 기능 카드 3개
- 데스크톱: 3열 Grid
- 태블릿: 2열 Grid
- 모바일: 1열 세로 배치
- 카드마다 아이콘, 제목, 설명, 링크 버튼 포함
```

---

### 2.3 요소별 스타일 분석

요소별로 다음 정보를 확인한다.

#### 텍스트

- 글꼴
- 글자 크기
- 굵기
- 줄 높이
- 자간
- 색상
- 줄바꿈 규칙
- 최대 너비
- 텍스트 정렬

#### 레이아웃

- 너비
- 높이
- 최소/최대 크기
- 내부 여백
- 외부 여백
- 요소 간 간격
- 정렬 방식
- Flex 또는 Grid 구조
- 화면 크기별 변화

#### 시각 요소

- 배경색
- 테두리
- 모서리 둥글기
- 그림자
- 투명도
- 블러
- 그라데이션
- 이미지 비율
- 아이콘 크기 및 정렬
- 적용 예정 클래스명 또는 스타일 책임 위치

#### 인터랙션

- Hover 상태
- Focus 상태
- Active 상태
- Disabled 상태
- 열림/닫힘 상태
- 탭 전환
- 드롭다운
- 모달
- 애니메이션
- 스크롤 연동 동작

---

### 2.4 공통 컴포넌트 분석

반복되는 요소를 식별하고 재사용 범위를 결정한다.

다음 기준으로 공통 컴포넌트화를 검토한다.

- 동일한 구조가 두 번 이상 반복되는가
- 스타일과 인터랙션이 일관되는가
- 향후 다른 페이지에서도 사용할 가능성이 있는가
- 컴포넌트화해도 API가 과도하게 복잡해지지 않는가
- 현재 프로젝트에 대체 가능한 기존 컴포넌트가 있는가

공통 컴포넌트 후보는 `TODO.md`에 기록한다.

단, `TODO.md`에는 단순히 공통 컴포넌트 이름만 적지 말고, 어떤 요소를 그 컴포넌트로 처리할지와 어떤 스타일 책임이 남는지도 같이 기록한다.

예시:

```md
- SectionHeading
- PrimaryButton
- FeatureCard
- IconTextList
- PageContainer
- ResponsiveGrid

- `SectionHeading`: `Our Mission`, `Our Partner`, `Our Business`, `Our Office`에 공통 적용
- `PrimaryButton`: Hero CTA와 Footer CTA에 재사용, about 전용 스타일 오버라이드 없음
```

---

### 2.5 기존 구현물 분석

새로운 코드 작성 전에 프로젝트 내부에서 다음 항목을 검색한다.

- root 혹은 body 등에 정의된 전역 공통 스타일 및 css 변수 (globals.css 파일 참고)
- 전역으로 정의된 스타일
- 동일한 페이지 또는 유사한 페이지
- 유사한 레이아웃
- 동일한 헤더 또는 푸터
- 기존 카드 컴포넌트
- 기존 버튼 컴포넌트
- 기존 모달 또는 탭 컴포넌트
- 색상 및 타이포그래피 토큰
- 반응형 유틸리티
- 아이콘 리소스
- 이미지 리소스
- 기존 API 타입 및 호출 코드

동일하거나 유사한 구현이 존재하는 경우 다음 중 하나를 선택한다.

1. 기존 컴포넌트를 그대로 재사용한다.
2. 기존 컴포넌트를 확장한다. 확장 시 어떤 부분을 변경하였는지 기록/보고한다.
3. 기존 컴포넌트의 구조가 부적절한 경우 새 컴포넌트를 만들되, 중복 이유를 기록한다.

---

## 3. 구현 계획 작성

분석 완료 후 구체적인 구현 단위 중심으로 TODO 목록을 작성한다.

TODO 항목은 실제로 구현할 대상이어야 한다.

계획 작성이 끝나면 즉시 구현을 시작하지 않는다. 다음을 사용자에게 먼저 공유하고 확인을 받는다.

- 구현 대상 범위
- 섹션별 작업 순서
- 각 요소에 적용할 주요 스타일 규칙
- 재사용할 공통 컴포넌트와 신규 작성이 필요한 부분
- 확인이 필요한 가정, 제외 범위, 미결 사항

사용자 확인 전에는 계획 수립 상태까지만 정리하고, 구현 단계의 TODO를 완료 처리하지 않는다.

### 올바른 TODO 예시

```md
- [ ] 공통 페이지 컨테이너 적용
- [ ] 상단 네비게이션 구현
- [ ] Hero 섹션 구현
- [ ] 기능 카드 컴포넌트 구현
- [ ] 기능 카드 목록 연결
- [ ] CTA 배너 구현
- [ ] 모바일 Grid 전환 구현
- [ ] Hover 및 Focus 상태 구현
- [ ] 데스크톱 시각 검증
- [ ] 모바일 시각 검증
```

### 잘못된 TODO 예시

```md
- [ ] 화면 분석
- [ ] 개발 계획 수립
- [ ] CSS 작성
- [ ] 코드 작성
- [ ] 수정하기
```

TODO 항목은 다음 조건을 만족해야 한다.

- 완료 여부를 명확히 판별할 수 있어야 한다.
- 실제 구현 대상 또는 검증 대상을 포함해야 한다.
- 너무 큰 단위라면 구현 가능한 단위로 분리해야 한다.
- 너무 작은 스타일 수정만으로 과도하게 세분화하지 않는다.
- 구현 순서가 의존성을 반영해야 한다.
- 각 TODO만 읽어도 어떤 대상의 어떤 수정인지 식별 가능해야 한다.

예시:

```md
- [ ] `styles.sectionTitle` 기준으로 `Our Mission`~`Our Office` 섹션 헤더 타이포 정리
- [ ] 인증 카드 4종의 이미지 높이, 카드 패딩, 설명 타이포를 원본 기준으로 보정
- [ ] `Our Business` 카드 썸네일 비율과 `Learn More` 버튼 크기 보정
```

---

## 4. 구현

계획된 TODO 순서에 따라 구현을 진행한다.

각 구현 단계가 완료되면 해당 TODO 항목을 즉시 완료 처리한다.

구현 단계가 끝났는데 `TODO.md`가 갱신되지 않은 상태로 다음 단계로 넘어가면 안 된다.

구현은 사용자에게 공유한 최신 계획을 기준으로 진행한다. 사용자가 계획 내용을 수정하거나 범위를 조정하면, 구현 전에 먼저 `TODO.md`와 계획 내용을 갱신한다.

### 4.1 구현 우선순위

일반적으로 다음 순서를 우선한다.

1. 페이지 진입 경로 및 기본 레이아웃
2. 페이지 전체 컨테이너와 반응형 구조
3. Header 및 Footer
4. 핵심 섹션 구조
5. 공통 컴포넌트
6. 카드, 리스트, 텍스트 등 세부 콘텐츠
7. 이미지 및 아이콘
8. 인터랙션
9. Hover, Focus, Disabled 상태
10. 반응형 세부 조정
11. 접근성 및 코드 정리

단, 특정 섹션이 다른 영역의 레이아웃 기준을 결정하는 경우 해당 섹션을 먼저 구현할 수 있다.

---

### 4.2 구현 시 준수 사항

- 프로젝트의 기존 폴더 구조와 명명 규칙을 따른다.
- 기존 디자인 토큰, CSS 변수, 공통 컴포넌트를 우선 사용한다.
- 동일한 스타일 값을 여러 위치에 중복 작성하지 않는다.
- 페이지 전용 스타일과 공통 스타일의 책임을 분리한다.
- 하드코딩된 위치값으로 반응형 구조를 억지로 맞추지 않는다.
- 이미지 비율과 잘림 방식을 참고자료에 맞춘다.
- 불필요한 절대 위치 지정은 피한다.
- 시각적 구조는 CSS Grid 또는 Flexbox를 우선 사용한다.
- 기존 구현과 충돌할 수 있는 전역 스타일 변경은 최소화한다.
- 작업 범위와 무관한 대규모 리팩터링은 수행하지 않는다.
- API 또는 데이터가 아직 준비되지 않은 경우에는 명확한 mock 데이터 또는 placeholder를 사용하고, 이를 TODO에 기록한다.
- 구현 후에는 `TODO.md`에 적용한 클래스명, 컴포넌트명, 스타일 수치를 즉시 반영한다.
- 변경 내역은 `A 컴포넌트의 font-size를 18px에서 16px로 조정`처럼 수정 전/후가 드러나게 기록한다.
- `더 자연스럽게`, `더 비슷하게`, `더 맞췄다`처럼 수치가 없는 표현만 단독으로 쓰지 않는다.

---

### 4.3 폰트 관련 주의 사항

텍스트가 참고 화면과 다르게 보이는 경우 단순히 `font-size`만 수정하지 않는다.

다음 항목을 함께 확인한다.

- 실제 로드된 font file
- `font-family` 우선순위
- fallback font 적용 여부
- `font-weight`
- Variable Font의 `font-variation-settings`
- `font-synthesis`
- `line-height`
- `letter-spacing`
- 브라우저 기본 스타일 상속 여부
- 폰트 로딩 실패 여부

동일한 `font-size`라도 실제 적용 글꼴과 굵기가 다르면 줄바꿈, 요소 높이, 여백 인상이 달라질 수 있다.

---

## 5. 검증

구현 완료 후 참고자료와 구현 결과를 비교하여 구조와 스타일이 일치하는지 검증한다.

검증은 단순한 육안 확인으로 끝내지 않는다.

검증이 끝난 뒤에는 `무엇을 확인했는지`뿐 아니라 `확인 결과가 어땠는지`도 `TODO.md`에 남겨야 한다.

### 5.1 구조 검증

다음을 확인한다.

- 섹션의 종류와 순서가 동일한가
- 각 섹션의 하위 요소가 누락되지 않았는가
- 카드, 버튼, 탭, 입력 요소의 개수가 동일한가
- 제목, 설명, 라벨, 보조 텍스트가 누락되지 않았는가
- Header, Footer, 고정 요소가 동일하게 존재하는가
- 모바일에서 요소 순서가 의도대로 변경되는가

---

### 5.2 스타일 검증

중요한 요소별로 원본과 구현 결과의 computed style을 비교한다.

최소 비교 대상은 다음과 같다.

```text
font-family
font-size
font-weight
line-height
letter-spacing
color
background-color
width
max-width
height
padding
margin
gap
border
border-radius
box-shadow
display
position
align-items
justify-content
grid-template-columns
```

특히 다음 값은 원본과 정확히 일치시키는 것을 우선 목표로 한다.

- `font-size`
- `font-weight`
- `line-height`
- `padding`
- `margin`
- `gap`
- `width`
- `max-width`

---

### 5.3 반응형 검증

참고자료에 모바일 또는 태블릿 화면이 존재하는 경우 반드시 해당 화면도 검증한다.

다음 항목을 확인한다.

- 컨테이너 좌우 여백
- 텍스트 줄바꿈
- Grid 열 수 변화
- 카드 폭과 높이
- 버튼 크기
- 이미지 비율
- 숨김 또는 노출되는 요소
- Header 메뉴 동작
- 가로 스크롤 발생 여부
- 터치 영역 크기
- 고정 요소와 콘텐츠의 겹침 여부

참고자료에 특정 브레이크포인트가 명시되어 있지 않다면, 프로젝트의 기존 브레이크포인트를 우선 사용한다.

---

### 5.4 인터랙션 검증

인터랙션이 포함된 화면이라면 다음 상태를 확인한다.

- 기본 상태
- Hover 상태
- Focus 상태
- Active 상태
- Disabled 상태
- 선택 상태
- 열림 상태
- 닫힘 상태
- 로딩 상태
- 빈 상태
- 오류 상태

마우스 기반 UI뿐 아니라 키보드 탐색과 포커스 이동도 확인한다.

확인이 힘들다면 무엇을 확인해야 하는지 목록을 작성해서 사용자에게 전달한다.

---

### 5.5 검증 결과 기록

검증 후 `TODO.md`의 `검증 결과` 섹션에 다음을 기록한다.

- 확인한 화면 크기
- 비교한 주요 요소
- 수정한 불일치 항목
- 아직 남아 있는 차이
- 차이가 발생한 이유
- 참고자료에서 확인할 수 없었던 항목
- 사용자 확인이 필요한 항목
- 각 검증 항목의 결과값
- 수정 전/후 비교 수치
- 검증에 사용한 클래스, 요소, 컴포넌트 식별자

예시:

```md
## 검증 결과

- 데스크톱 1440px 기준 검증 완료
- 모바일 390px 기준 검증 완료
- Hero 제목: `styles.heroTitle`, `font-size 60px -> 64px`, `line-height 76px -> 84px`, `max-width 960px -> 900px`
- 기능 카드: `styles.featureCard`, `padding 32px 28px -> 24px 20px`, `gap 28px -> 20px`
- 원본 이미지 리소스를 확보하지 못해 대체 이미지 사용
- 모바일 메뉴의 실제 열림 애니메이션은 참고자료에서 확인 불가
```

---

## 6. 사용자 검토

구현 및 내부 검증이 완료되면 작업 상태를 `사용자 검토 중`으로 변경한다.

사용자에게 다음 내용을 명확히 전달한다.

- 구현한 페이지 또는 섹션
- 반영한 주요 요소
- 재사용하거나 새로 만든 공통 컴포넌트
- 확인이 필요한 가정 또는 차이점
- 남아 있는 제한 사항
- 검토가 필요한 화면 크기 또는 기능

사용자가 수정 요청을 하면 해당 요청의 성격에 따라 다음 단계를 반복한다.

- 구조 변경: 분석 → 계획 → 구현 → 검증
- 스타일 변경: 분석 → 구현 → 검증
- 인터랙션 변경: 분석 → 계획 → 구현 → 검증
- 공통 컴포넌트 변경: 영향 범위 분석 → 구현 → 회귀 검증

수정 요청이 이전 분석 결과와 충돌하는 경우 최신 사용자 요청을 우선한다.

## 7. 완료 처리

사용자가 구현 결과에 대해 명시적으로 승인한 경우에만 작업을 완료 처리한다.

완료 시 다음을 수행한다.

1. `TODO.md`의 미완료 항목을 확인한다.
2. 완료된 구현 항목을 모두 체크 처리한다.
3. 남은 제한 사항 또는 보류 항목을 별도로 기록한다.
4. 작업 상태를 `완료`로 변경한다.
5. 구현 결과와 변경 파일을 정리한다.
6. 권장되는 후속 작업이 있다면 제안한다.

### 완료 요약 형식

```md
## 완료 요약

- 구현 페이지:
- 구현 경로:
- 신규 컴포넌트:
- 재사용 컴포넌트:
- 반영한 반응형 화면:
- 검증 완료 항목:
- 남은 제한 사항:
- 사용자 승인 일시:
```

### 권장 가능한 후속 작업 예시

- 실제 API 데이터 연결
- 로딩, 빈 상태, 오류 상태 추가
- 접근성 개선
- 키보드 탐색 보완
- 이미지 최적화
- 성능 측정
- E2E 테스트 추가
- 시각 회귀 테스트 추가
- 다국어 대응
- 다크 모드 대응

사용자 승인 없이 임의로 완료 상태로 변경하지 않는다.

---

## 금지 사항

다음과 같은 작업 방식은 피한다.

- 참고자료를 충분히 분석하지 않고 바로 구현을 시작하는 행위
- 이미지 한 장만 보고 수치를 과도하게 추정하는 행위
- 기존 공통 컴포넌트를 확인하지 않고 동일한 UI를 중복 구현하는 행위
- 코드 선언값만 확인하고 실제 렌더링 값을 검증하지 않는 행위
- 데스크톱 화면만 구현하고 모바일 화면을 무시하는 행위
- 완료되지 않은 TODO를 완료 처리하는 행위
- 선행조건이 완료되지 않았는데 다음 상세 TODO를 진행하는 행위
- 구현 계획을 세우지 않았거나, 계획 내용을 사용자에게 확인받지 않은 상태에서 구현을 시작하는 행위
- 사용자 승인 전에 작업을 완료 처리하는 행위
- 구현 범위와 무관한 대규모 리팩터링을 수행하는 행위
- 확인되지 않은 인터랙션을 사실처럼 단정하는 행위
- 폰트, 이미지, 아이콘 라이선스 또는 사용 권한을 확인하지 않고 외부 리소스를 무단 복제하는 행위
- 변경 내용을 구체적 수치 없이 모호한 표현으로만 기록하는 행위
- 어떤 요소를 구현했는지, 어떤 스타일 규칙을 적용했는지 `TODO.md`에 남기지 않는 행위

---

## 작업 완료 기준

다음 조건을 충족해야 구현 완료 후보로 판단한다.

- 참고자료 기준의 섹션 구조가 구현되어 있다.
- 필요한 요소의 종류와 개수가 일치한다.
- 공통 요소가 적절히 재사용되거나 컴포넌트화되어 있다.
- 주요 텍스트 스타일과 레이아웃 값이 검증되었다.
- 주요 화면 크기에서 레이아웃이 정상 동작한다.
- 주요 인터랙션 상태가 구현되었다.
- 확인된 불일치 항목이 기록되어 있다.
- `TODO.md`가 최신 상태로 정리되어 있다.
- 사용자의 최종 검토 및 승인이 완료되었다.
