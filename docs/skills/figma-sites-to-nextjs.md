# Figma Sites To Next.js

## 목적

- 제공된 원본 사이트, 주로 Figma Sites 공개 페이지를 분석해 현재 프로젝트의 화면에 반영한다.
- 디자인 변경이 들어오면 변경된 화면 요소를 파악해 기존 구현에 안전하게 반영한다.
- 기본적으로 화면 디자인만 분석하고 수정한다.
- 디자인 변경으로 인해 특정 기능이나 로직이 불필요해지거나 새로 필요해지는 경우에는 직접 변경하지 않고 사용자에게 알린다.

## 방법

- Codex의 `@chrome` 또는 브라우저 검사 도구를 사용해 원본 사이트와 localhost를 조사한다. 이 중에서 `@chrome`을 사용하는 것을 기본으로 하고 만약 불가하다면 사용자에게 어떤 방법으로 사이트들을 조사할 것인지 후보군을 제안하도록 한다.
- 이 workflow를 사용할 때는 `docs/skills/figma-sites-to-nextjs.md`를 따른다고 명시한다.
- 화면 분석은 section 단위를 기준으로 진행한다. section은 같은 레벨의 다른 블록과 명확히 구분되며, 동일 레벨의 다른 section과 위치를 바꿔도 의미나 구조가 깨지지 않는 단위를 뜻한다.
- section 안에 더 작은 subsection이 있을 수 있다. 다만 처음부터 자잘한 subsection으로 쪼개지 말고, 큰 section의 padding, margin, 높이, 정렬 등 외곽 레이아웃을 먼저 맞춘 뒤 내부 subsection으로 내려가고, 마지막에 다시 큰 section으로 올라와 전체 형태가 유지되는지 재확인한다.
- 원본과 타겟 페이지의 section 구조를 각각 먼저 파악하고 대응 관계를 만든다.
- 원본에는 있고 타겟에는 없는 section은 추가 대상으로 분류한다.
- 타겟에는 있고 원본에는 없는 section은 바로 삭제하지 말고, 제거 후보로 보고한 뒤 주석 처리 등 되돌릴 수 있는 방식으로만 조치한다.
- 두 화면에 공통으로 존재하는 section은 section 단위 스크린샷, 요소 측정, 텍스트/자산 검증을 이용해 차이를 확인하고 보정한다.
- Figma Sites의 DOM 구조, CSS, JavaScript를 최종 프로젝트 코드로는 그대로 복사하지 않는다. 다만 1단계 기준 HTML 생성에서는 원본을 최대한 직접적으로 가져오는 것을 우선하며, 이후 단계에서만 프로젝트 구조에 맞게 정리한다.
- 원본 DOM은 조작하지 않는다. 다만 예외적으로 이미지 등 리소스에 `loading="lazy"`가 적용되어 있어 비교, 분석, 스크린샷 정확도를 해친다면 해당 속성은 제거한 뒤 분석 및 스크린샷 촬영을 진행할 수 있다.
- 최종 결과물은 현재 프로젝트 구조에 맞는 React / Next.js 코드여야 한다.
- 단계를 진행할 때마다 무엇을 하였고 어떤 중간 산출물이 나왔는지 보고한다. 이후 확인을 받고 다음 단계로 진행한다.
- 기준 HTML은 `file://` 경로로 직접 여는 방식에 의존하지 않는다. 로컬에서 브라우저로 확인할 수 있도록 node 기반 로컬 서버에서 접근 가능한 위치에 생성하거나 복사해 둔다.
- 원본 사이트 또는 로컬 페이지를 분석하거나 스크린샷을 찍기 전에는 로딩 시간을 고려해 최소 5초간 대기한 뒤 수행한다.
- 브라우저의 `fullPage` 스크린샷은 환경에 따라 같은 viewport가 반복 캡처되는 오류가 날 수 있으므로, 캡처 후 반드시 스크롤 높이와 이미지 결과를 검증한다. 반복 캡처 정황이 있으면 즉시 분할 스크린샷 방식으로 다시 촬영한다.
- 비교용 임시 산출물은 workspace root 바로 아래에 흩어놓지 말고, 항상 `.temp/` 아래에 작업별 하위 폴더를 만들어 저장한다.
- 여러 viewport를 오가며 비교할 때는 viewport를 섞지 않는다. 특정 viewport 하나를 정한 뒤, 그 viewport에서 `원본 분석 -> 로컬 비교 -> 수정 -> 재검증`까지 완료하고 나서만 다음 viewport로 이동한다.
- 같은 turn 안에서도 데스크톱과 모바일처럼 서로 다른 viewport의 측정값이나 스크린샷을 한 비교 묶음 안에 섞지 않는다.

## 전체 흐름

이 작업은 아래 3단계를 기준으로 진행한다.

1. 기준 HTML 생성
2. 기준 HTML 정리 및 구조화
3. 기존 코드에 변경사항 반영

핵심 원칙은 다음과 같다.

- 먼저 원본을 분석하기 쉬운 순수 HTML 기준물로 만든다.
- 그 다음 기준 HTML을 바탕으로 디자인은 유지하면서 더 효율적인 코드 구조로 정리한다.
- 마지막으로 정리된 결과와 기존 코드를 비교해 실제 변경이 필요한 화면 요소만 반영한다.
- 디자인과 무관한 로직, 상태, 라우팅, 데이터 처리 코드는 임의로 수정하지 않는다.

## 1. 기준 HTML 생성

### 목표

- 원본 사이트의 화면을 분석하기 쉬운 정적 기준물로 만든다.
- 이 단계의 목표는 프로젝트에 바로 반영할 최종 코드 작성이 아니라, 원본 구조와 스타일을 최대한 유지한 기준 HTML을 먼저 확보하는 것이다.

### 작업 방식

- 우선 원본 사이트의 요소들에 접근해서 전체 내용을 그대로 가지고 와서 기준 HTML을 만든다. 이후 분석과 비교를 어렵게 만드는 요소만 최소한으로 정리한다.
    - 우선적으로 불필요한 스크립트를 제거하거나 비활성화해 스크립트 의존성이 없는 정적 HTML 상태로 만든다.
    - 동적으로 생성되는 클래스 기반 스타일은 일반적인 정적 스타일 구조로 변환한다.
    - 난독화되었거나 의미를 알기 어려운 class명은 적당히 의미 있는 이름으로 바꾼다. 다만 class명을 지나치게 잘게 나누거나, 아직 확인되지 않은 컴포넌트 구조까지 미리 추상화하지는 않는다.
- 이 단계의 목적은 보기 좋은 코드 작성이 아니라, 원본에 최대한 가까운 분석용 기준물을 안정적으로 확보하는 것이다.
- 생성한 기준 HTML은 이후 브라우저 비교와 스크린샷 검증이 가능하도록 node 기반 로컬 서버에서 접근 가능한 위치에 둔다.
- font-family에 대해 "Pretendard:Bold"와 같이 지정된 경우 `font-weight: bold`와 같은 형태로 변경해 적용하도록 한다. `SemiBold: 600, Medium: 500, regular: 생략` 형태로 적용한다.

### 이 단계에서 확인할 것

- section 순서
- 전체 레이아웃 구조
- 컨테이너 폭
- 좌우 padding
- section 간격
- typography
- 이미지 크기와 crop
- 버튼 크기와 정렬
- background / overlay / gradient
- desktop, tablet, mobile에서의 기본 배치

### 주의사항

- 이 단계의 HTML은 최종 산출물이 아니라 분석용 기준물이다.
- 원본 사이트의 JavaScript 동작까지 유지하려고 하지는 않는다.
- 다만 처음부터 화면을 눈으로 보고 다시 작성하려고 하지는 않는다.
- 먼저 원본을 최대한 그대로 가져온 뒤, 분석에 방해되는 부분만 정리한다.
- 생성형 className은 의미 있는 이름으로 바꿀 수 있지만, 시각 비교가 어려워질 정도로 과하게 구조화하지 않는다.
- 스크립트를 제거하거나 class명을 바꾸는 과정에서 시각 결과가 달라지지 않도록 주의한다.

## 2. 기준 HTML 정리 및 구조화

### 목표

- 1단계에서 만든 기준 HTML을 바탕으로, 시각 결과는 유지하면서 더 효율적인 코드 구조로 정리한다.
- 반복되는 스타일, 불필요한 중복, 과한 개별 보정값을 줄여 재사용 가능한 구조로 바꾼다.

### 작업 방식

- 기준 HTML을 분석해 반복되는 section 패턴, 카드 패턴, CTA 패턴, 공통 container 규칙을 찾는다.
- 상위 컴포넌트나 공통 스타일에서 관리할 수 있는 값은 끌어올린다.
- 의미 있는 section 단위와 컴포넌트 단위로 구조를 나눈다.
- 임시 보정값이 꼭 필요하지 않다면 제거한다.
- 정리 과정에서 디자인 결과가 달라져서는 안 된다.

### 이 단계에서 정리할 대상

- 반복되는 spacing
- 반복되는 typography 규칙
- 중복된 button 스타일
- 공통 container 폭과 padding
- 반복되는 카드 레이아웃
- section별 공통 구조
- 불필요한 개별 margin / padding 보정

### 주의사항

- 보기 좋은 코드로 바꾸는 것이 목적이지, 디자인을 새로 해석하는 것이 목적은 아니다.
- 구조화 과정에서 원본과 다른 줄바꿈, 간격, 이미지 crop이 생기지 않도록 주의한다.
- 이 단계에서도 아직 기존 프로젝트 코드에는 바로 덮어쓰지 않는다.
- 먼저 분석용 결과물을 안정적인 구현 기준으로 정리하는 데 집중한다.

## 3. 기존 코드에 변경사항 반영

### 목표

- 2단계에서 정리된 구현 기준과 기존 프로젝트 코드를 비교해 실제 변경이 필요한 부분만 반영한다.
- 기존 코드의 로직과 구조를 최대한 보존하면서 화면 변경만 적용한다.

### 작업 방식

- 정리된 기준 코드와 현재 프로젝트의 section 구조를 대응시킨다.
- 어떤 section이 동일하고, 어떤 section이 달라졌는지 먼저 구분한다.
- 누락된 요소는 추가한다.
- 제거된 요소는 사용자에게 보고한다.
- 변경된 요소는 영향 범위를 보고 반영한다.

### 바로 반영해도 되는 변경

- 단일 section 내부의 색상 변경
- 구조 변화 없는 spacing 조정
- 구조 변화 없는 typography 조정
- 이미지 교체 또는 crop 조정
- 버튼 스타일 조정
- 배경, border, radius, shadow 조정

### 사용자 확인 후 진행할 변경

- section 구조가 바뀌는 경우
- 컴포넌트 분리가 새로 필요한 경우
- 기존 인터랙션과 충돌 가능성이 있는 경우
- 기존 기능 또는 로직의 삭제 가능성이 있는 경우
- 새로운 상태 처리나 이벤트 연결이 필요한 경우
- 단순 스타일 수정으로 보기 어려운 규모의 UI 변경인 경우

### 주의사항

- 디자인과 무관한 로직을 수정하거나 삭제하지 않는다.
- UI 변경 때문에 불필요해진 로직이 보이면 사용자에게 알린다.
- 기존 정상 영역까지 함께 리팩터링하지 않는다.
- 변경 범위 밖의 section을 같이 손보지 않는다.

## 권장 작업 순서

실제 작업은 아래 순서를 권장한다.

1. 원본 사이트와 localhost를 열어 비교 기준을 맞춘다.
2. 이번 비교에 사용할 viewport를 하나 정하고, 원본과 localhost 모두 그 viewport에서만 작업한다.
3. 각 페이지가 열린 뒤 로딩 시간을 고려해 최소 5초간 대기한다.
4. 원본 사이트를 기준으로 분석용 순수 HTML 파일을 만든다.
5. 원본과 비교해 기준 HTML의 시각 일치도를 먼저 맞춘다.
6. 기준 HTML을 더 효율적인 구조로 정리한다.
7. 정리된 결과를 기존 프로젝트의 각 section과 비교한다.
8. 큰 section의 외곽 레이아웃을 먼저 맞춘다.
9. 이후 필요한 경우 그 section 내부 subsection 단위로 내려가 세부 차이를 맞춘다.
10. subsection 수정 후 다시 큰 section 단위로 올라와 전체 형태가 유지되는지 검증한다.
11. 실제 변경이 필요한 부분만 기존 코드에 반영한다.
12. 반영 후 localhost에서 다시 시각 검증한다. 이때도 분석 및 스크린샷 전 최소 5초간 대기한다.
13. 현재 viewport에서 작업이 끝난 뒤에만 다음 viewport로 이동한다.

## 완료 검증

- 원본 사이트와 localhost를 같은 viewport에서 다시 비교한다.
- 반영한 section이 원본 의도와 다르지 않은지 확인한다.
- 구조화 과정에서 디자인이 달라지지 않았는지 확인한다.
- 기존 로직이나 기능이 의도치 않게 수정되지 않았는지 확인한다.
- `tsc`와 `lint`를 실행해 코드 오류를 확인한다.
- 스크린샷 비교가 필요할 때는 원본과 로컬 모두 5초 이상 로딩 대기 후 캡처했는지 확인한다.
- 전체 페이지 스크린샷을 사용했다면, 문서 높이와 실제 캡처 결과가 맞는지 검증한다. 같은 화면이 반복되거나 특정 구간이 누락되면 해당 캡처는 폐기하고 다시 촬영한다.
- 사용자가 최종 결과를 승인하면 `.temp/` 아래 이번 작업에 사용한 비교용 임시 파일은 삭제한다.

## 스크린샷 비교 원칙

- 우선 같은 viewport에서 원본과 localhost의 `window.innerWidth`, `window.innerHeight`, `document.documentElement.scrollHeight`를 함께 기록한다.
- `fullPage` 캡처를 사용했다면 캡처 이미지가 문서 전체를 반영하는지 직접 확인한다. 동일한 hero나 동일한 section이 반복되면 실패로 간주한다.
- 비교는 기본적으로 분할 캡처 방식으로 진행한다. `fullPage` 캡처는 보조 참고용으로만 쓰고, 최종 비교 근거로는 section 단위 또는 분할 캡처 결과를 우선한다.
- viewport 고정 높이 기준으로 기계적으로 나누기보다, 먼저 section 경계와 핵심 콘텐츠 블록 위치를 파악한 뒤 section 단위로 스크린샷을 찍어 비교한다.
- section 하나가 viewport보다 크면 해당 section 안에서 다시 여러 장으로 나눠 촬영한다.
- 실패 가능성이 조금이라도 보이면 viewport 높이 단위로 스크롤 위치를 나눠 분할 캡처한다.
- 분할 캡처 시에는 각 구간마다 `window.scrollTo(0, y)` 후 150ms 이상 안정화 시간을 두고 촬영한다.
- 분할 기준은 `0`, `1 * viewportHeight`, `2 * viewportHeight`처럼 고정 간격만 쓰지 말고, 마지막 구간이 반드시 문서 하단을 포함하도록 `maxScrollY`도 포함한다.
- section 단위 비교 시에는 각 section의 시작점, 높이, 촬영 범위를 함께 기록한다.
- 비교용 파일명에는 viewport와 스크롤 위치 또는 section 이름을 포함한다. 예: `remote-1440-hero-y0.png`, `local-1440-comparison-y900.png`.
- sticky 요소나 비디오처럼 시간에 따라 달라지는 구간은 정적 비교 대상과 분리해서 본다. 필요하면 해당 section 전후를 따로 캡처한다.
- 브라우저가 background tab 스크린샷을 잘못 반환할 가능성이 있으면, 비교 대상마다 같은 단일 탭에서 `원본 촬영 -> URL 검증 -> 저장 -> 로컬 촬영 -> URL 검증 -> 저장` 순서로 다시 진행한다.
- 각 캡처 파일과 함께 촬영 시점의 `tab id`, `URL`, `title`, `scrollY`를 메타데이터로 남겨, 잘못된 탭이 찍혔는지 사후 검증할 수 있게 한다.
- 작업별 임시 파일은 `.temp/<task-name>/` 아래에 저장하고, 스크린샷, 메타데이터, 비교 로그를 같은 하위 폴더에 함께 둔다.
- 최종 보고 시에는 어떤 캡처 방식으로 검증했는지 함께 남긴다. 예: `1440px 분할 8장`, `390px 분할 11장`.

검증 결과 처리 기준은 아래와 같다.

- error가 있으면 수정한다.
- 바로 수정할 수 없는 error는 사용자에게 알린다.
- 새롭게 작성한 코드에서 발생한 warning은 가능한 범위에서 정리한다.
- 기존 코드에서 이미 존재하던 warning은 구분해서 보고한다.

## 임시 파일 정리

- 기준 HTML 등 임시 산출물은 작업 중에는 유지할 수 있다.
- 임시 산출물은 `.temp/` 아래에만 둔다.
- 최종 반영과 검증이 끝나면 삭제 여부를 정리해 보고한다.
- 사용자가 최종 결과를 승인하면 이번 작업에 한해 생성한 `.temp/<task-name>/` 하위 파일은 삭제한다.
- 사용자가 검토를 위해 유지하길 원하면 남겨둘 수 있다.
