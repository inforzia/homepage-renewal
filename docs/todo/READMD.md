## 개요

본 폴더에는 향후 수행해야 하거나 현재 진행 중인 작업의 계획, 진행 상태, 사용자 확인 사항, 사전조사 결과를 기록하는 TODO 문서를 작성합니다.

단순히 한두 단계로 즉시 처리할 수 있고 별도 계획, 검증, 사용자 확인이 필요하지 않은 작업은 TODO 문서를 생성하지 않아도 됩니다.

다음 중 하나에 해당하는 작업은 원칙적으로 TODO 문서를 생성합니다.

- 여러 단계의 구현 또는 수정이 필요한 작업
- 작업 중 의사결정, 사용자 확인 또는 방향성 합의가 필요한 작업
- 여러 파일, 화면, 기능 또는 시스템에 영향을 주는 작업
- 테스트, 검증 또는 배포 확인이 필요한 작업
- 이후 재개하거나 다른 작업자가 이어서 진행할 가능성이 있는 작업
- 사용자가 명시적으로 todo를 생성하거나 계획 후 작업을 수행하라고 지시한 작업

TODO 문서는 작업 단위별로 하나씩 생성합니다.

파일명은 다음 형식을 사용합니다.

```text
YYYY-MM-DD-작업-요약-todo.md
```

예시:

```text
2026-06-30-user-profile-edit-todo.md
2026-06-30-payment-refund-api-todo.md
```

완료된 TODO 문서는 `/completed` 폴더로 이동합니다.

---

## 문서 구조

TODO 문서는 원칙적으로 다음 순서로 작성합니다.

```md
# 작업 제목

## 메타 정보

- 요청자:
- 생성 일시:
- 최종 수정 일시:
- 완료 일시:
- 상태: 계획 중 | 진행 중 | 검증 중 | 사용자 확인 대기 | 완료 | 보류
- 관련 작업:
- 관련 문서 / 이슈 / 링크:

## 목적

- 이 작업을 수행해야 하는 이유
- 완료되었을 때 기대되는 결과

## 상세지시

- 범위를 한정하거나 제외해야 하는 조건
- 이번 작업에서 따를 구현/운영/언어 정책
- 이번 작업에서 의도적으로 하지 않을 내용

## 사전조사 자료

### ...

- 관련 TODO:
- 참고 자료:

## 완료 기준

- 기능, 화면, 문서 또는 산출물이 충족해야 하는 조건
- 사용자 확인이 필요한 기준
- 특별한 검증이 필요한 경우 그 기준

## TODO 목록

- [ ] [1. ...](#1-...)
- [ ] [2. ...](#2-...)
- [ ] [3. ...](#3-...)

## 상세 TODO

### 1. ...

- 상위 TODO: [TODO 목록으로 이동](#todo-목록)
- 목적:
- 작업 내용:
    - [ ] ...
    - [ ] ...
- 선행조건:
    - [ ] [선행 TODO](#...)
- 참고 자료:
- 검증 방법:

## 사용자 확인 필요 사항

- [ ] 사용자 확인 항목
- [ ] 사용자 결정이 필요한 항목
- [ ] 배포, 운영 또는 디자인 확인이 필요한 항목

## 작업 결과 요약

- 변경된 주요 내용
- 남은 제약 또는 후속 작업
```

## 작성 원칙

### 1. 메타 정보

문서 최상단에는 작업의 식별과 추적에 필요한 메타 정보를 작성합니다.

반드시 포함해야 하는 항목은 다음과 같습니다.

- 요청자
- 생성 일시
- 최종 수정 일시
- 상태
- 완료 일시
- 관련 문서, 이슈 또는 작업 링크

완료 일시는 실제로 모든 작업과 검증이 끝난 시점에만 작성합니다.

---

### 2. 목적 상세지시와 사전조사 자료

`목적`에는 무엇을 구현하거나 수정하는지가 아니라, 왜 해당 작업이 필요한지와 어떤 결과를 만들어야 하는지를 작성합니다.

`상세지시`는 `목적` 바로 아래에 배치합니다.

여기에는 다음과 같은 내용을 기록합니다.

- 범위를 한정하는 조건
- 제외해야 하는 작업
- 사용해야 하는 언어/locale/정책/전략
- 사용자 요청으로 확정된 구현 방향

예를 들면 다음과 같은 내용이 들어갈 수 있습니다.

- locale 작업은 한글/영어만 고려한다.
- 실제 번역 문구 작성은 이번 작업에서 수행하지 않는다.
- 기존 URL은 유지하고 문구만 locale 대응한다.

`사전조사 자료`는 TODO 목록 위에 배치합니다.

여기에는 다음과 같은 내용을 기록합니다.

- 현재 구현 현황을 보여주는 코드/문서/링크
- 기존 코드 구조를 파악하는 데 필요한 참고 파일
- 조사 결과를 뒷받침하는 자료
- 외부 문서, 디자인, 명세, 참고 링크

즉, `사전조사 자료`에는 오로지 참고자료만 적습니다. 설명, 판단, 결정 이유, 미결 사항은 이 섹션에 적지 않습니다.

---

### 3. 완료 기준

각 작업에는 명확한 완료 기준이 있어야 합니다.

완료 기준에는 가능한 한 다음 내용을 포함합니다.

- 구현 완료 조건
- 사용자 확인이 필요한 조건
- 특별한 검증이나 환경 확인이 필요한 조건

단순히 코드가 작성되었다는 이유만으로 TODO를 완료 처리하지 않습니다.

다만 일반적이고 당연한 검증 절차를 완료 기준에 반복해서 장황하게 적을 필요는 없습니다. 특별한 검증 포인트가 있는 경우에만 적습니다.

---

### 4. TODO 목록 작성 순서

TODO 목록은 다음 원칙을 우선순위로 하여 정렬합니다.

1. 작업 순서상 선행되어야 하는 항목을 먼저 작성합니다.
2. 의존성이 있는 항목은 인접하게 배치합니다.
3. 동일한 기능 영역 또는 작업 분류에 속하는 항목은 가능한 한 함께 배치합니다.
4. 화면 작업은 사용자 화면 흐름 또는 표시 순서를 고려합니다.
5. CRUD 작업은 일반적으로 Create, Read, Update, Delete 순서를 따릅니다.
6. 구현 이후 사용자 확인, 후속 보완 순서가 자연스럽게 이어지도록 구성합니다.

TODO 항목은 가능한 한 실행 가능한 단위로 작성합니다.

좋지 않은 예시:

```md
- [ ] 사용자 관리 기능 구현
```

좋은 예시:

```md
- [ ] 사용자 목록 조회 API 연동
- [ ] 사용자 목록 화면 구현
- [ ] 사용자 상세 조회 화면 구현
- [ ] 사용자 정보 수정 API 연동
- [ ] 사용자 정보 수정 화면 반영
```

추가로, TODO 작성 시점에 아직 결정되지 않은 사항이 있다면 다음 원칙을 따릅니다.

- 간단하고 바로 답을 받아 결정할 수 있는 항목이면 TODO 작성 전에 사용자에게 질의하여 결정하고 들어갑니다.
- 결정까지 시간이 오래 걸리거나, 지금 당장 결정할 수 없고, 바로 실행할 작업이 아니라면 TODO 제목에 `(결정 필요)`를 붙여 남깁니다.

예시:

```md
- [ ] locale 적용 언어 범위 결정
- [ ] locale URL 정책 정리 (결정 필요)
```

### 5. 상세 TODO 작성 규칙

상세 설명이 필요한 TODO는 `상세 TODO` 섹션에서 확장합니다.

상세 TODO에는 기본적으로 다음을 기록합니다.

- 목적
- 작업 내용
- 선행조건 (필요시)
- 참고 자료 (필요시)
- 검증 방법 (특별한 경우에만)
- 검증 결과 (검증을 수행한 경우)

다음 규칙을 반드시 따릅니다.

- `완료 조건` 항목은 사용하지 않습니다. 의미가 모호하고 다른 항목과 중복되기 쉽기 때문입니다.
- `작업 내용`은 설명문이 아니라 task 목록 형태로 작성합니다.
- `작업 내용`의 각 항목은 그 자체로 작업이거나, 완료 확인을 위해 통과해야 하는 조건이어야 합니다.
- `작업 내용`의 각 항목은 어떤 대상 화면, 컴포넌트, 클래스, API, 문서, 데이터 등을 구현/수정/검증하는지 명시해야 합니다.
- 스타일 작업이라면 가능하면 클래스명 또는 컴포넌트명과 함께 핵심 스타일 규칙을 적습니다.
- `작업 내용`의 단일 항목이 별도 계획이 필요할 정도로 크다면, 그 항목은 새로운 TODO 또는 하위 상세 TODO로 분리하고 서로 링크를 연결합니다.
- `선행조건`은 말 그대로 해당 작업을 진행하기 전에 반드시 먼저 완료되어야 하는 작업이 있을 때만 작성합니다.
- 단순히 같이 관련되어 있거나 어느 쪽을 먼저 해도 되는 작업은 `선행조건`으로 쓰지 않습니다.
- `선행조건`에는 모호한 설명 대신 먼저 완료되어야 하는 실제 작업 링크를 작성합니다.
- 어떤 작업이 `선행조건`으로 사용된다면, 그 작업은 반드시 해당 작업보다 위의 TODO에 배치합니다.
- `선행조건`이 있는 상세 TODO는 선행 작업이 실제로 완료 check 된 뒤에만 진행할 수 있습니다.
- 선행조건이 완료되었는데 check가 반영되지 않았다면, 먼저 해당 선행 작업의 상태와 결과를 문서에 반영한 뒤 다음 작업으로 이동합니다.
- 아직 결정되지 않은 항목을 상세 TODO에 남겨야 한다면, 해당 TODO 제목이나 작업 항목에 `(결정 필요)`를 붙여 나중에 작업 방향을 정해야 함을 명시합니다.
- `검증 방법`은 다음과 같은 경우에만 작성합니다.
    - 사용자가 명시적으로 특정 확인 방법을 지시한 경우
    - 검증용 도구가 명시적으로 제공된 경우
    - 일반적인 구현 확인만으로 부족하고 특별한 확인 절차가 필요한 경우
- lint, typecheck, tsc, build 같은 일반적이고 당연한 검증은 개별 상세 TODO의 `검증 방법`에 반복해서 적지 않습니다.
- 특별한 검증이 필요 없고 구현 여부만 확인하면 되는 작업은 `검증 방법`을 생략할 수 있습니다.
- 검증을 수행했다면 `검증 결과` 또는 동등한 항목에 실제 결과를 기록합니다. 단순히 `검증함`이라고만 쓰지 않습니다.
- 변경 결과는 `더 비슷하게 수정`처럼 모호하게 쓰지 않고, 가능하면 `font-size 18px -> 16px`, `padding 32px -> 24px`처럼 전후 차이가 드러나게 기록합니다.

### 6. 사전조사 자료와 TODO의 역할 분리

현황을 보여주는 참고자료 자체는 `사전조사 자료`에 적습니다.

TODO 목록과 상세 TODO에는 실제로 수행해야 하는 작업만 적습니다.

검증 현황이나 결과를 일일이 별도 로그처럼 남기지 않습니다. 작업 중 발견된 문제는 다음 방식으로 반영합니다.

- 검증 중 추가 보완이 필요하면 새로운 task를 TODO에 추가합니다.
- 구현 과정에서 드러난 제약이나 미결 사항은 `상세지시`, `(결정 필요)` TODO, 또는 `사용자 확인 필요 사항`으로 옮깁니다.

즉, TODO의 task들이 작업과 보완의 기록 역할을 함께 수행합니다.

다만, 실제로 수행한 검증의 결과값과 핵심 차이점은 생략하지 않습니다. 나중에 문서를 다시 볼 때 `무엇을 확인했고 결과가 무엇이었는지`를 알 수 있어야 합니다.

---

### 7. 불확실한 작업의 처리

TODO 작성 시점에 작업 범위나 구현 방식이 완전히 확정되지 않은 경우, 추측으로 세부 계획을 확정하지 않습니다.

이 경우 다음 순서로 처리합니다.

1. 현재 확정된 정보만 기준으로 포괄적인 TODO를 작성합니다.
2. 불확실한 부분은 `상세지시`, `(결정 필요)` TODO, 또는 `사용자 확인 필요 사항`에 분리합니다.
3. 가능한 선택지와 각 선택지의 영향, 장단점, 권장안을 함께 정리합니다.
4. 작업 진행 중 구체화된 내용이 생기면 TODO를 갱신하고 하위 task를 추가합니다.
5. 사용자 결정이 반드시 필요한 경우에만 사용자 확인을 요청합니다.

사용자 확인이 필요하지 않고 합리적인 기본값이나 권장안이 존재하는 경우에는, 해당 가정을 문서에 기록한 뒤 작업을 진행합니다.

## 활용 방법

### 1. 작업 시작

새 작업 지시를 받으면 먼저 작업의 복잡도와 계획 필요 여부를 판단합니다.

별도 계획이 필요한 작업이라면 TODO 문서를 생성하고 다음을 먼저 작성합니다.

- 목적
- 상세지시
- 사전조사 자료
- 완료 기준
- 상위 TODO 목록
- 사용자 확인이 필요한 미결 사항

하위 계획이 명확한 항목은 즉시 상세 TODO로 확장합니다.

사용자 확인이 필요한 항목은 성급하게 세부 구현 계획으로 확정하지 않고 포괄적인 TODO로 유지합니다.

또한, 계획 기반으로 진행하는 작업은 TODO 초안과 상세 TODO를 정리한 뒤 곧바로 구현하지 않습니다. 우선 계획 내용을 사용자에게 공유하고, 작업 시작 전 확인을 받은 다음 구현 단계로 진입합니다.

---

### 2. 작업 진행

작업은 TODO 순서와 의존성을 기준으로 진행합니다.

이때 구현, 수정, 검증 같은 실제 작업 착수는 사용자에게 계획 내용을 공유하고 확인받은 뒤에만 진행합니다. 사용자 확인 이전에는 분석, 자료 정리, 계획 보강까지만 수행합니다.

상세화되지 않은 TODO를 진행해야 하는 시점에는 다음 중 하나를 수행합니다.

- 기존 문서, 코드, 요구사항을 분석하여 계획을 구체화합니다.
- 합리적인 구현안이 있는 경우 권장안을 제시하고 문서에 기록합니다.
- 사용자 결정이 필수적인 경우 선택지와 영향 범위를 정리하여 확인받습니다.

작업 중 새 요구사항이나 변경 사항이 발생하면 영향 범위를 확인합니다.

기존 작업 범위 안에 포함되는 변경은 TODO를 갱신합니다.

범위가 크게 달라지거나 독립적인 목표가 생기는 경우에는 별도 TODO 문서를 생성하고 관련 작업으로 연결합니다.

특정 작업이 완료되면 어떤 작업을 완료했는지 사용자에게 명시적으로 보고하여 사용자가 중간 경과를 확인할 수 있도록 합니다. 작업이 완료되면 완료된 작업은 todo 파일에 check 표시로 반영되어 있어야 합니다. 다만 사용자의 확인이 필요한 경우 해당 작업을 멋대로 완료처리하지는 않으며, 사용자가 최종 판단에서 결과를 승인하지 않고 보완조치를 요구하거나 일부 혹은 전체를 롤백하라고 지시하면 해당 todo는 check가 해제되어야 합니다.

상세 TODO 하나가 완료되었거나, 구현/검증 대상을 다른 상세 TODO로 옮기는 시점마다 TODO 문서를 즉시 갱신합니다. 여러 상세 TODO를 처리한 뒤 한꺼번에 몰아서 갱신하지 않습니다.

---

### 3. 완료 처리

각 TODO 항목은 다음 조건을 모두 만족한 경우에만 완료 표시합니다.

- 작업 내용이 실제로 반영되었음
- 해당 항목에 특별한 검증 방법이 있었다면 그 확인이 수행되었음
- 알려진 제한 사항이 있다면 문서에 반영되었음
- 선행 또는 연관 TODO와의 상태가 모순되지 않음

완료된 항목은 다음 형식으로 표시합니다.

```md
- [x] 사용자 목록 조회 화면 구현
```

검증 실패, 보류 또는 사용자 확인 대기 상태의 항목은 완료 표시하지 않습니다.

---

### 4. 사용자 확인 및 최종 완료

최종적으로 사용자가 직접 확인해야 하는 사항은 문서 마지막의 `사용자 확인 필요 사항`에 정리합니다.

사용자 확인 항목에는 가능한 한 다음을 포함합니다.

- 확인해야 하는 화면 또는 기능
- 사용자의 결정이 필요한 선택지
- 남아 있는 제한 사항 또는 후속 작업

사용자가 보완 사항을 요청하면 관련 TODO를 추가하거나 기존 TODO를 확장하여 작업을 진행합니다.

사용자가 모든 항목이 완료되었다고 판단하거나, 최종 완료 처리를 지시하면 다음을 수행합니다.

1. `작업 결과 요약`을 작성합니다.
2. 완료 일시와 최종 상태를 기록합니다.
3. TODO 문서를 `/completed` 폴더로 이동합니다.
4. 관련 작업 문서가 있다면 완료 상태를 반영합니다.
